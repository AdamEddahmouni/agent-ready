# ADR-0009: Pre-1.0 stability and compatibility policy

## Status

Accepted

## Context

Agent-Ready is pre-1.0. Contributors and early adopters need explicit
expectations about what can change between releases, since "pre-1.0, may
change" is too vague to be useful in practice.

## Alternatives considered

- No stated policy before 1.0 (common but leaves adopters guessing).
- Full SemVer compatibility guarantees starting now (premature — the
  project hasn't yet had real-world usage to validate the shape of the
  contract, schema, or diagnostics).
- A tiered policy: some surfaces frozen early, others explicitly
  changeable, documented per-surface.

## Decision

Adopt a tiered, explicitly documented policy (see also `README.md`'s
"Stability" section and `docs/specification/schema-versioning.md`):

- **JSON Schema (`schemas/v1/agent-ready.schema.json`)**: the _shape_
  described by contract `version: 1` will not have required fields added
  or existing fields removed/retyped without a major version bump of the
  schema itself (i.e. a `version: 2` contract and, if warranted, a
  `schemas/v2/` file). Additive optional fields may be added in minor
  releases of the Agent-Ready package while contract `version: 1` remains
  current.
- **Diagnostic codes**: the codes in `src/diagnostics/codes.ts` are
  stable identifiers. New codes may be added; existing codes will not be
  renamed or repurposed to mean something materially different. Human
  message _text_ (`summary`, `detail`, `remediation`) is explicitly **not**
  covered by any compatibility guarantee and may be reworded at any time.
- **Normalized contract shape (`NormalizedContract`)**: covered by the
  same policy as the JSON Schema it derives from — additive changes are
  minor-version-safe; shape changes to existing fields are breaking.
- **CLI output**: `--json` output shape follows the same additive-only
  policy as diagnostics and the normalized contract. Human-readable
  (non-JSON) CLI output is explicitly **not** covered by any stability
  guarantee — it is designed for readability, not machine parsing, and
  may be reformatted at any time; scripts must use `--json`.
- **Public programmatic API (`src/index.ts`)**: everything exported from
  this entry point is public but explicitly **experimental** before 1.0 —
  shapes may change between minor versions, with changes noted in
  `CHANGELOG.md`. Anything not exported from this entry point (ajv
  internals, CLI argument-parsing internals, etc.) carries no
  compatibility guarantee at all, at any version.
- **Exit codes**: the 5-value scheme in ADR-0008 is considered stable pre-1.0
  as well, since shell scripts are especially sensitive to silent exit-code
  changes.

## Consequences

- Contributors know which changes require a changelog entry flagged as
  breaking versus a routine addition.
- Early adopters can script against `--json` output and exit codes with
  reasonable confidence, while being warned off relying on human-readable
  text or un-exported internals.

## Reconsideration trigger

Revisit this entire policy when approaching 1.0: at that point, the
"experimental" qualifier on the public API should be dropped (or the API
surface trimmed further first), and full SemVer guarantees should apply.
